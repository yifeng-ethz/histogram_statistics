# histogram_statistics_v2 Wave Reporting

This directory hosts the Phase-4 waveform reporting package for the standalone
`histogram_statistics_v2` UVM harness.

## Artifact Tree

- `cases/<case-id>/meta.json`: machine-readable metadata for one published waveform case
- `cases/<case-id>/summary.md`: short human-readable summary for that case
- `cases/<case-id>/wavedrom.json`: curated WaveDrom summary input
- `cases/<case-id>/wavedrom.svg`: rendered WaveDrom summary
- `generated/`: local VCD artifacts generated by the harness and ignored by git
- `templates/`: reusable GTKWave save-file templates
- `manifest.json`: generated aggregate index consumed by `index.html`
- `index.html`: static browser entry point for published wave cases

## Promoted Publication Set

The initial publication set is intentionally small:

- `HIST-SMOKE`: directed UVM bring-up reference
- `P141-REAL-TRIAD`: promoted long-run real-hit coverage anchor

These two cases cover the minimum useful publication surface:

- deterministic configuration/apply plus first bank-swap readout
- long-run multi-phase real-hit traffic with queue activity and plateau-backed promotion

Additional `P`, `E`, and `X` families should only be published when they add a
new explanation, failure signature, or interaction pattern.

## Marker Convention

Major boundaries use `T0`, `T1`, `T2`, ... and phase-specific markers use
semantic names such as `cfg_apply`, `first_hit`, `bank_swap`, `REAL_A`,
`REAL_B`, and `REAL_C`.

Recommended baseline markers:

- `rst_deassert`
- `cfg_apply`
- `first_hit`
- `bank_swap`
- `burst_read`
- `queue_peak`
- `drain_done`

## Signal Group Conventions

The checked-in GTKWave template separates:

- clock/reset and interval control
- CSR/bin-readout activity
- ingress/snoop activity
- coalescing queue and divider pipeline state
- bank-swap/readout progress

Use decimal for counters and occupancy, hex for payload vectors, and analog
duplicates for queue fill-level style signals when a case-specific save file is
curated.

## Minimal Workflow

1. Generate a VCD with `uvm/scripts/run_vcd_case.sh` or the underlying `make run_vcd` target.
2. Create or update a case package with `uvm/scripts/publish_wave_case.py`.
3. Generate `manifest.json` with `make -C uvm wave_index`.
4. Open `waves/index.html` in a browser to inspect the published package.

The current implementation provides the directory layout, metadata/index
automation, a wrapper script, and a reusable GTKWave template. Per-case `.gtkw`
curation is still a manual follow-up when a case needs presentation polish.
